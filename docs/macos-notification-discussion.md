## 历史问题与最终结论

### 1. 首次通知权限与业务提醒发送耦合

历史现象：
- 首次触发通知时，系统先弹出通知权限授权。
- 用户感知像是“第一条提醒没有成功送达”。

最终结论：
- 已将“请求授权”和“发送业务提醒”拆开。
- 当前不在真正到点发送通知时申请权限。
- 当前是在“让 notify reminder 生效”的前端动作上做预检：
  - 新建 notify reminder
  - 启用 notify reminder
  - 把启用中的 rest reminder 切成 notify
- reminder 的保存接口不再因为当前通知不可用而阻止配置保存。

### 2. 通知标题未统一按有效语言展示

历史现象：
- 中文场景下通知标题可能出现英文 `Reminder`。

最终结论：
- 已修复。
- 通知标题统一走“有效语言”结果，不再直接按原始 `settings.UI.Language` 分支。
- 当前行为：
  - 中文标题：`提醒`
  - 英文标题：`Reminder`

### 3. 菜单栏应用形态与通知相关激活行为

讨论背景：
- 之前担心点击通知或系统激活路径会把 Pause 拉成普通 App 形态，出现 Dock 图标或窗口激活异常。

当前结论：
- 构建层已通过受版本管理的链路写入 `LSUIElement=1`。
- 普通业务通知点击回调当前由 `UNUserNotificationCenterDelegate` 接管，但回调中不主动执行窗口展示或激活动作。
- 如后续需要“点击通知后做某个具体产品动作”，应单独设计，而不是在当前回调里直接恢复窗口行为。

### 4. 首次授权时点击“拒绝”或进入系统设置会导致应用退出

历史现象：
- 在首次安装后启用 notify reminder。
- 系统弹出通知授权。
- 用户点击“拒绝”或进入系统通知设置后，Pause 会退出。

最终结论：
- 该问题已在 2026-03-27 修复。
- 根因不在前端，也不在 `app.Quit()` / `Shutdown()` 链路。
- 根因集中在旧的 macOS 通知授权“同步 cgo + semaphore 等待结果”桥接实现。
- 修复方式是改成“Objective-C 异步回调 -> Go channel 等待”的桥接。
- 修复后，deny 场景会稳定回到 `denied` 状态，不再触发进程退出。

需要特别记录的系统行为：
- macOS unified log 在 deny 场景下仍会出现：
  - `Requested authorization [ didGrant: 0 hasError: 1 hasCompletionHandler: 1 ]`
- 这现在被视为系统返回特征之一，而不是应用崩溃本身。

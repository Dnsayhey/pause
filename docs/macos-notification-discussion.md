# macOS 通知问题讨论记录

更新时间：2026-03-26

## 问题 1：首次通知权限与提醒触发耦合

### 现象
- 在首次触发通知时，系统先出现通知权限授权流程。
- 用户感知上像是“第一条提醒没有成功送达”。

### 根因（产品层面）
- macOS 的系统通知需要用户授权后才能展示。
- 如果把“请求授权”和“业务提醒发送”放在同一个触发点，首次体验会不稳定。

### 已确认方案
- 将“请求授权”与“发送提醒”拆分为两个动作。
- 在用户第一次创建`通知型提醒`时触发权限申请（而不是在第一次真正到点提醒时申请）。
- 授权成功后不主动发送“测试通知”。
- 若用户首次拒绝授权：
  - 后续不再反复弹系统授权；
  - 在产品内提示用户前往系统设置手动开启通知权限。

### 交互约定
- 未决（Not Determined）：可在应用内先展示一次权限说明，再触发系统授权。
- 已拒绝（Denied）：显示引导文案和“打开系统设置”入口。
- 已授权（Authorized）：正常发送业务提醒通知。

## 问题 2：通知标题文案没有按中文展示

### 现象
- 通知中出现标题 `Reminder`，内容是提醒名（例如“喝水”）。
- 期望是标题“提醒”，内容“喝水”。

### 根因（实现层面）
- 项目里已有多语言实现，但通知标题这条链路对 `auto` 语言值处理不完整。
- 通知标题在 runtime engine 内按 `settings.UI.Language` 直接判断。
- 当该值为 `auto` 时，通知路径未统一走“有效语言解析”，可能落到英文标题。

### 方案选项
- 选项 A：在通知路径中复用统一的“有效语言解析”（推荐）。
- 选项 B：在通知路径做本地兜底（如按 locale / 提醒名字符集猜测语言）。

### 推荐方案
- 采用选项 A：通知标题统一使用“有效语言”结果，避免和 UI 语言策略分叉。
- 中文有效语言时标题固定为“提醒”；英文有效语言时标题为“Reminder”。

## 问题 3：点击通知后出现 Dock 图标

### 现象
- 点击系统通知后，Dock 出现 Pause 图标。
- 期望是保持菜单栏应用形态，不在 Dock 出现图标。

### 根因（实现与构建链路）
- 当前主要依赖运行时激活策略控制（Accessory），对“通知点击激活”这类系统路径不够稳。
- 另外，`build` 目录下的 plist 不是版本追踪源，直接修改该文件无法形成稳定方案。

### 已确认原则
- 不在未追踪的构建产物文件中做长期修复。
- Dock 策略应落在“受版本管理且稳定进入构建链路”的配置/模板层。

### 方案选项
- 选项 A：在受管 plist 模板声明 `LSUIElement=1`，并保留运行时策略兜底（推荐）。
- 选项 B：仅依赖运行时激活策略，不改构建模板（不推荐，稳定性较弱）。

### 关于通知点击事件
- 可以接管通知点击回调（`UNUserNotificationCenterDelegate` / action）。
- 可控目标：点击后不自动打开主窗口、仅执行应用内部动作（如聚焦状态栏逻辑）。
- 约束：系统点击通知仍可能触发应用激活流程，因此需要与 Dock/激活策略联动设计。

### 最终确认方案（问题 3）
- 仅采用两层处理：
  - 构建层：在受版本管理且进入产物的构建链路中设置 `LSUIElement=1`。
  - 通知点击层：接管通知点击回调，但回调中不执行任何窗口展示/激活动作（即“啥也不干”）。
- 明确不采用：
  - 修改 `build/` 目录下未追踪的 plist 文件本体。
  - 额外引入运行时兜底策略（当前阶段先不做）。

## 补充：macOS / Windows 通知状态说明

### macOS 可能状态（UNUserNotificationCenter）
- `NotDetermined`：尚未请求过通知权限。
- `Denied`：用户拒绝过，应用侧通常不能再次触发系统授权弹窗。
- `Authorized`：已授权。
- `Provisional`：临时授权（macOS 场景相对少见）。

说明：
- 即使是 `Authorized`，系统层展示能力（例如横幅/声音）仍可能被用户单独关闭，最终展示效果可能受影响。

### Windows 侧状态特征
- Windows 对桌面应用不提供与 macOS 完全对等的“首次授权状态机”。
- 更接近“通知能力/可达性状态”，常见场景包括：
  - 系统全局通知关闭；
  - 应用级通知关闭；
  - 专注助手/勿扰/展示模式压制；
  - 策略（组策略）限制；
  - 通知通道可用。

### Windows 可获取范围（实现可行性）
- 可通过 `ToastNotifier.Setting` 获取主要开关态，例如：
  - `Enabled`
  - `DisabledForApplication`
  - `DisabledForUser`
  - `DisabledByGroupPolicy`
  - `DisabledForManifest`
- 可选通过 `SHQueryUserNotificationState` 判断当前是否处于 `QUIET_TIME`、`PRESENTATION_MODE`、全屏等可能压制通知的环境。

限制：
- 无法得到与 macOS `NotDetermined` 等价的 per-app 首次授权状态。
- 只能判断“当前可通知能力”，不能 100% 保证每条通知都以横幅等形式展示。
